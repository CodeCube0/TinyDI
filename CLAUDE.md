# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

TinyDI is a minimal, type-safe, decorator-free Dependency Injection container for TypeScript ("Explicit over magic") — no reflection, no decorators, no automatic constructor injection, zero runtime dependencies. The core library (`src/`) is complete; `examples/` and the root `README.md`/`README.it.md` (Phase 2) are done; `docs-app/`, the documentation/project site (Phase 3), is done; governance, CI/CD, npm-publish readiness, and `ROADMAP.md` (Phase 4) are done. The blog (Phase 5) is not yet implemented — see "Repository layout notes" below. The package is published on npm as **`tinydi-container`** (not `tinydi` — see the npm package name note further down). `0.1.1` was published manually (the account's npm 2FA mode requires a one-time interactive browser approval on every publish, which a non-interactive CI job can't complete); `0.1.2` was published fully automatically once `release.yml` was switched to npm Trusted Publishing (OIDC) — see the CI/release note below for how that's wired up.

## Commands

```bash
npm run build          # tsup -> dist/ (ESM + CJS + .d.ts/.d.cts, dual package)
npm test                # vitest run (all tests)
npm run test:watch      # vitest (watch mode)
npm run test:coverage   # vitest run --coverage (v8, thresholds: 95% lines/functions/branches/statements)
npm run lint            # eslint .
npm run lint:fix
npm run format           # prettier --write .
npm run format:check
npm run docs:api        # typedoc -> api-docs/ (gitignored, standalone JSDoc-derived API reference)
npm run changeset       # record a changeset for the current change (see CONTRIBUTING.md)
```

Run a single test file: `npx vitest run tests/container.test.ts`
Run a single test by name: `npx vitest run -t "circular dependency"`

Build is bundled with `tsup` (`tsup.config.ts`, entry `src/index.ts`, uses `tsconfig.build.json`), emitting `dist/index.js` (ESM), `dist/index.cjs` (CJS), and matching `.d.ts`/`.d.cts` — a genuine dual-package build, not a straight per-file `tsc` emit. `tsconfig.build.json` sets `"ignoreDeprecations": "6.0"` because TypeScript 6.0 hard-errors (`TS5101`) on the `baseUrl` option that `tsup`'s declaration-bundling step synthesizes internally; this is a tsup/TS6 interaction, not something in our own tsconfig.

### Running an example

Every folder under `examples/` is an independent package that depends on the core via `"tinydi-container": "file:../.."` (not an npm workspace) — build the core first, then install/run the example:

```bash
npm run build                 # from the repo root, once (or after any src/ change)
cd examples/<name>
npm install                   # re-run after core changes too, to refresh the file: dependency
npm start                     # basic, service, repository-pattern, node-backend (tsx scripts)
npm run dev                   # vue, nuxt, react (dev server)
```

`examples/**` is excluded from the root `eslint.config.js` and has no lint config of its own — each example only type-checks via its own `tsconfig.json` (`vue-tsc` for the Vue example, plain `tsc` for React/Node ones, Nuxt's generated `.nuxt/tsconfig.json` for Nuxt). Each example's `package-lock.json` is committed, same as the root's (see below), so the exact dependency set that was verified to work stays reproducible.

### Running the docs site

`docs-app/` is a separate package (own `package.json`), unrelated to the `tinydi-container` npm package itself — it doesn't depend on `src/` or `dist/` at all, it's a static site _about_ the library.

```bash
cd docs-app
npm install   # also runs postinstall: copies self-hosted font files from node_modules
npm run build # node build.mjs — src/content + src/templates -> dist/ (20 static HTML pages)
npm run serve # node serve.mjs — tiny zero-dependency static server at :4550
npm run dev   # build + serve
```

`docs-app/dist` is generated output (gitignored, like the core's `dist/`) — never hand-edit files there, edit `docs-app/src/` and rebuild. `docs-app/src/assets/fonts/*.woff2` are also gitignored (regenerated by the `postinstall` script from the `@fontsource-variable/*` packages). `docs-app/**` is excluded from the root `eslint.config.js` for the same reason `examples/**` is.

### CI, release, and governance

- **`.github/workflows/ci.yml`** — runs on every push to `main` and every PR: `lint` job (ESLint + Prettier check, single Node version), `test` job (`npm run test:coverage`, matrixed across Node 18/20/22 — 18 is the `engines` floor, 22 is current), `build` job (`npm run build`).
- **`.github/workflows/commitlint.yml`** — validates PR commit messages against Conventional Commits (`commitlint.config.js` extends `@commitlint/config-conventional`) using `wagoid/commitlint-github-action`. This is CI-only by design — there is deliberately no local `husky` git hook, so it never blocks a local commit, only a PR.
- **`.github/workflows/release.yml`** — on push to `main`, runs `changesets/action`: if there are pending changesets it opens/updates a "Version Packages" PR (bumps version, updates `CHANGELOG.md`); merging that PR triggers the same workflow to actually `npm publish` (via the root `release` script, which runs `build` then `changeset publish`), with npm provenance (`permissions.id-token: write` + `publishConfig.provenance: true` in `package.json`). The very first publish (`tinydi-container@0.1.1`) had to be done manually from an authenticated local session, because the npm account's 2FA mode requires a one-time interactive browser approval (`EOTP`) on every publish — a plain `NPM_TOKEN`, regardless of token type/scope, cannot satisfy that from a non-interactive CI job (this is also what blocked the first CI attempts, on top of an unrelated bug where the token was passed under the wrong env var name, `NODE_AUTH_TOKEN` instead of the `NPM_TOKEN` name `changesets/action` actually reads). The workflow now uses **npm Trusted Publishing** (OIDC) instead of a token: the `NPM_TOKEN` env var was removed from the `changesets/action` step (so it can't fall back to token auth), the job's `node-version` is 22, and an explicit `npm install -g npm@latest` step runs before `npm ci` (Trusted Publishing needs npm CLI ≥11.5.1, which the bundled version may not satisfy). This requires a matching **Trusted Publisher entry configured on `npmjs.com/package/tinydi-container/access`** (GitHub Actions → org `CodeCube0`, repo `TinyDI`, workflow filename `release.yml`) — done manually on npmjs.com, not something a repo file can express. **Verified working**: `tinydi-container@0.1.2` published fully automatically through this path (Version Packages PR merged → `release.yml` ran → real `npm publish`, no manual OTP step). The publish path only actually runs when there's a pending changeset for a not-yet-published version; pushing to `main` with nothing to release just no-ops past the `changesets/action` step (as seen between 0.1.1 and 0.1.2).
- **Changesets** (`.changeset/config.json`, `.changeset/*.md` pending-change files) is the single source of truth for versioning and `CHANGELOG.md` — see `CONTRIBUTING.md`'s "Changesets (versioning)" section for the contributor-facing flow (`npx changeset`). The repo is single-package (no npm workspaces), so changesets only ever version the root `tinydi-container` package; `examples/**` and `docs-app/**` are untouched by it.
- **`typedoc.json`** generates a standalone API reference from the same JSDoc already required on every public member (`npm run docs:api` → `api-docs/`, gitignored). It is intentionally _not_ wired into `docs-app`'s own hand-built static-site generator/design system — that would mean reconciling two very different HTML outputs (TypeDoc's own theme vs. `docs-app`'s Vite-like dark design) for a phase-4 bonus item; it's a separate, unstyled reference output for now.
- Governance docs (`CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, `CHANGELOG.md`) and GitHub templates (`.github/ISSUE_TEMPLATE/*.yml`, `.github/pull_request_template.md`) follow the same "no real domain/contact yet" placeholder pattern as `docs-app`'s `SITE_URL` (`https://tinydi.dev`) — e.g. `CODE_OF_CONDUCT.md`'s enforcement contact uses the same placeholder domain, and `SECURITY.md` routes reports through GitHub's private vulnerability reporting instead of an email address, to avoid inventing one.

## Architecture

Everything lives in `src/`, one concern per file, wired together in `src/index.ts` (the public API surface — anything not exported there is internal):

- **`token.ts`** — `createToken<T>(description)` returns a `Token<T>` keyed by a `symbol` (identity), not by its `description` string (which is only a debug label — two tokens with the same description are distinct registrations). The type `T` is carried via an optional `__type?: T` phantom property that is never actually assigned at runtime; this is what lets `container.resolve(token)` infer the return type with no explicit generic. When touching `Token`, preserve this: don't add a real runtime-assigned field for `T`.
- **`service-lifetime.ts`** — `ServiceLifetime` enum, only `Singleton` (default) and `Transient`. **Do not add `Scoped` here** — it's an intentional future extension, not in scope for the current version (see `task/01_core_library.md` for the reasoning).
- **`errors.ts`** — error hierarchy: abstract `ContainerError` (base, carries the involved `Token`) → `RegistrationError`, `ResolutionError`, `CircularDependencyError`. `CircularDependencyError` renders the full cycle path as `A\n -> B\n -> C\n -> A`; that exact format is asserted in tests, so if you touch `CircularDependencyError.formatPath`, keep the first token unprefixed and every subsequent one prefixed with `' -> '`, including the repeated closing token.
- **`container.ts`** — the `Container` class. Key internal invariants:
  - Registering the same token twice (`registerInstance` or `registerFactory`, in any combination) throws `RegistrationError`. This is a deliberate design choice (not obvious from the task spec) made to give `RegistrationError` an actual purpose — see `reports/01-core-library.md` for the rationale. To replace a registration, callers must `remove()` first (or `clear()`).
  - Factories have the signature `(container: Container) => T` — the container is passed explicitly to the factory (not captured via closure) so dependency wiring stays visible and so the container can track the resolution path.
  - Circular dependency detection works via an internal `resolutionPath: Token<unknown>[]` stack pushed/popped around each factory invocation; when a token already on the stack is requested again, it throws `CircularDependencyError` with the stack plus the repeated token.
  - Singleton factories cache their built instance on the internal registration record (`cachedInstance`/`hasCachedInstance`); Transient factories never cache.

`examples/` holds 7 independent demo packages (`basic`, `service`, `repository-pattern`, `node-backend`, `vue`, `nuxt`, `react`), each with its own `package.json`/`tsconfig.json`/README. In every framework example (`vue`, `nuxt`, `react`) the DI composition root (`di/container.ts` or `services/*.ts` + `plugins/di.ts`) stays framework-agnostic — it never imports Vue/Nuxt/React — and a thin, framework-idiomatic adapter (`di/plugin.ts` for Vue, `plugins/di.ts` + `composables/useService.ts` for Nuxt, `di/DIProvider.tsx` for React) is the only place that bridges the container into that framework's own DI-like mechanism (`provide`/`inject`, Nuxt plugin `provide`, React Context). Keep that separation when touching these examples.

`docs-app/` is a hand-written static-site generator (no 11ty/Astro/etc.) that turns structured content into the plain HTML/CSS/JS site actually shipped — see `docs-app/README.md` for the full file-by-file breakdown. The parts worth knowing before touching it:

- **Content is data, not markup.** Each doc page is `docs-app/src/content/{en,it}/docs/<id>.mjs`, exporting `meta` (title/description) and `blocks` — a small DSL (`heading`/`p`/`list`/`code`/`callout`/`api-table`/`compare-table`/`example-grid`/`faq-list`) rendered by `src/lib/blocks.mjs`. That same renderer extracts the search-index chunks (one per h2/h3/FAQ item) from the identical source, so content and search index can never drift apart. `docs-app/src/nav.mjs` is the one place that lists the 9 doc pages and every UI string per language — add a new doc page there first.
- **No shipped runtime dependency does syntax highlighting or search.** `src/lib/highlight.mjs` is a hand-rolled regex tokenizer (ts/bash/json) run at _build_ time, emitting already-colored `<span>`s baked into the static HTML (no client-side highlighting JS, no FOUC). `src/scripts/search.js` fetches the prebuilt `search-index.{en,it}.json` and scores matches with a small hand-written substring scorer — no Fuse.js or similar.
- **The scroll-spy ("On this page") is a plain scroll listener, not `IntersectionObserver`.** `src/scripts/nav.js` finds the last heading scrolled past (rAF-throttled), so a TOC entry stays active for its whole section rather than only while its heading is on screen. Also: `IntersectionObserver`'s `rootMargin` only accepts `px`/`%`, never `rem` — it throws a silent-in-the-UI `SyntaxError` if you get that wrong (this shipped broken once; only caught by checking the browser console, not by reading the code).
- **Language switching never auto-redirects.** The chosen language persists in `localStorage`, but a mismatched page only shows a dismissible banner ("Continue in English"/"Continua in italiano") — never a forced redirect (bad for deep links, back-navigation, and `hreflang`-evaluating crawlers). See `reports/03-docs-website.md` for the reasoning.
- **Any visual state that depends on `localStorage`/`sessionStorage`/`navigator.*` must be decided by an inline, synchronous script in `<head>`, before first paint** — never by a `defer`red script. `templates/layout.mjs` has three of these (`themeInitScript`, `platformInitScript`, `langBannerInitScript`), each setting a `data-*` attribute on `<html>` that CSS then reads (e.g. `:root[data-lang-banner-show] [data-lang-banner] { display: block; }`). This shipped broken once for the language banner: it used to be decided in the deferred `scripts/lang.js`, which only runs after the page has already rendered, so readers with a stored language preference saw the banner pop in and shove the whole page down a beat after every load/navigation (only caught by reproducing it in the browser with a stored preference set, not by reading the code). `scripts/lang.js` now only handles persisting the choice and the dismiss button.
- **The one piece of bespoke imagery** is `src/lib/graph-diagram.mjs`, a hand-built SVG of the actual resolve flow (Token → Container → Factory → Instance), reused unlabeled in the hero and labeled in the homepage Architecture section — deliberately not a stock photo.
- Every icon SVG needs an explicit size somewhere in `src/styles/components.css` (a class targeting it or its parent) — `base.css` has a defensive fallback (`svg { width: 1em; height: 1em; }`) precisely because a missing size rule once rendered a raw icon at its ~300×150 intrinsic size instead of failing loudly.

## Design decisions to preserve

- No dependency arrays in `package.json` — only `devDependencies` (tooling) and an optional `peerDependencies.typescript: ">=6.0.0"`. Don't introduce a runtime dependency without discussing it first; "zero external dependencies in the core" is a hard constraint from the project spec.
- Minimum supported TypeScript is **6.0** (`devDependencies.typescript` is pinned to `^6.0.3`, capped below the range `typescript-eslint@8.65.0` supports, which is `>=4.8.4 <6.1.0`; `typedoc@0.28.x`'s peer range, `5.0.x`–`6.0.x`, is the other current ceiling). If bumping TypeScript past 6.1, check both `typescript-eslint` and `typedoc` compatibility first.
- The published package is a genuine dual ESM+CJS build via `tsup` (not a bundler-free `tsc` emit) — `package.json`'s `exports` map has separate `import`/`require` conditions, each with their own `types` entry (`dist/index.d.ts` for ESM, `dist/index.d.cts` for CJS), matching what `tsup` actually emits. Keep both conditions in sync if the entry point ever changes.
- Every public API member needs full JSDoc (signature, params, return, a usage `@example`, relevant edge cases) — this is enforced by convention, not by a lint rule, so check for it in review.
- `eslint.config.js` and `vitest.config.ts` are deliberately excluded from typed linting (`*.config.js`/`*.config.ts` override with `disableTypeChecked`) because they aren't covered by `tsconfig.json`'s `include`. `tsconfig.json` (base, no `rootDir`) covers `src` + `tests` for editor/lint/type-check; `tsconfig.build.json` adds `rootDir: "src"` and excludes tests, and is what `npm run build` actually uses to emit `dist/`.
- `examples/**` is in the root `eslint.config.js` `ignores` list. Without it, `npm run lint`'s `projectService: true` tries to type-check every example's own `dist`/`.output`/`.nuxt` build artifacts too (they aren't covered by any `tsconfig.json`), producing hundreds of parse errors unrelated to the core.
- Coverage target is >95%; the current suite is at 100% on all `src/*.ts` (excluding `index.ts`, which is pure re-exports).
- The root `package-lock.json` is committed (as of Phase 4) — it used to be gitignored, but `.github/workflows/*.yml`'s `actions/setup-node@v4` (`cache: npm`) and every `npm ci` step need a real, checked-out lockfile to work at all; without one, every CI/release job fails immediately at the `setup-node` step. Same reproducibility rationale as `examples/*`'s and `docs-app`'s already-committed lockfiles.

## Repository layout notes

- `task/` and `reports/` exist locally but are **gitignored** (`/task`, `/reports` in `.gitignore`) — they hold the phased project spec and the end-of-phase reports and are intentionally not pushed to the remote. They still exist on disk in this working directory and are worth reading for context (e.g. `task/01_core_library.md`/`task/02_examples_and_readme.md`/`task/03_docs_website.md` for the original spec and constraints, `reports/01-core-library.md`/`reports/02-examples-and-readme.md`/`reports/03-docs-website.md` for what was actually built, problems hit, and scope trade-offs), but don't assume a fresh clone of the repo will have them.
- The task spec describes one further phase not yet implemented: an internal blog + closing write-up (`task/05_...`). The npm package is published as **`tinydi-container`**, not `tinydi` — the unscoped name `tinydi` is blocked by npm's anti-typosquatting policy for being too similar to the pre-existing package `tiny-di` (discovered only at actual publish time, as a `403`/`404` depending on auth state — not something `npm view tinydi` during Phase 4 planning could have caught, since that only checks for an exact existing name, not similarity). The project/brand name is still "TinyDI" everywhere (README, docs-app, GitHub repo); only the npm package name and its `npm install`/`import` specifier differ from it. `README.md`/`README.it.md`/`docs-app` all document `npm install tinydi-container` accordingly. `PRODUCT.md` and `DESIGN.md` at the repo root are the Impeccable-skill strategic/visual-design source of truth behind `docs-app/` (register: brand; Vite-like dark-first direction, one accent, self-hosted fonts) — read them before changing the site's visual language. `docs-app/src/nav.mjs`'s `SITE_URL` is a placeholder domain (`https://tinydi.dev`) used for `hreflang`/canonical/sitemap URLs until a real domain exists. `RELEASE.md` at the repo root is the maintainer-facing step-by-step runbook for cutting a release (changeset → Version Packages PR → merge → automated npm publish); keep it in sync with `release.yml` if that workflow changes.
