# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

TinyDI is a minimal, type-safe, decorator-free Dependency Injection container for TypeScript ("Explicit over magic") ŌĆö no reflection, no decorators, no automatic constructor injection, zero runtime dependencies. The core library (`src/`) is complete; `examples/` and the root `README.md`/`README.it.md` (Phase 2) are done; `docs-app/`, the documentation/project site (Phase 3), is done; governance, CI/CD, npm-publish readiness, and `ROADMAP.md` (Phase 4) are done. The blog (Phase 5) is not yet implemented ŌĆö see "Repository layout notes" below. The package has still never actually been published to npm ŌĆö Phase 4 wired up the release pipeline, but the first real `npm publish` (and adding the `NPM_TOKEN` repo secret it needs) is a separate, deliberately manual step.

## Commands

```bash
npm run build          # tsup -> dist/ (ESM + CJS + .d.ts/.d.cts, dual package)
npm test                # vitest run (all tests)
npm run test:watch      # vitest (watch mode)
npm run test:coverage   # vitest run --coverage (v8, thresholds: 95% lines/functions/branches/statements)
npm run lint            # eslint .
npm run lint:fix
npm run format           # prettier --write .
npm run format:check
npm run docs:api        # typedoc -> api-docs/ (gitignored, standalone JSDoc-derived API reference)
npm run changeset       # record a changeset for the current change (see CONTRIBUTING.md)
```

Run a single test file: `npx vitest run tests/container.test.ts`
Run a single test by name: `npx vitest run -t "circular dependency"`

Build is bundled with `tsup` (`tsup.config.ts`, entry `src/index.ts`, uses `tsconfig.build.json`), emitting `dist/index.js` (ESM), `dist/index.cjs` (CJS), and matching `.d.ts`/`.d.cts` ŌĆö a genuine dual-package build, not a straight per-file `tsc` emit. `tsconfig.build.json` sets `"ignoreDeprecations": "6.0"` because TypeScript 6.0 hard-errors (`TS5101`) on the `baseUrl` option that `tsup`'s declaration-bundling step synthesizes internally; this is a tsup/TS6 interaction, not something in our own tsconfig.

### Running an example

Every folder under `examples/` is an independent package that depends on the core via `"tinydi": "file:../.."` (not an npm workspace) ŌĆö build the core first, then install/run the example:

```bash
npm run build                 # from the repo root, once (or after any src/ change)
cd examples/<name>
npm install                   # re-run after core changes too, to refresh the file: dependency
npm start                     # basic, service, repository-pattern, node-backend (tsx scripts)
npm run dev                   # vue, nuxt, react (dev server)
```

`examples/**` is excluded from the root `eslint.config.js` and has no lint config of its own ŌĆö each example only type-checks via its own `tsconfig.json` (`vue-tsc` for the Vue example, plain `tsc` for React/Node ones, Nuxt's generated `.nuxt/tsconfig.json` for Nuxt). Each example's `package-lock.json` is committed, same as the root's (see below), so the exact dependency set that was verified to work stays reproducible.

### Running the docs site

`docs-app/` is a separate package (own `package.json`), unrelated to the `tinydi` npm package itself ŌĆö it doesn't depend on `src/` or `dist/` at all, it's a static site _about_ the library.

```bash
cd docs-app
npm install   # also runs postinstall: copies self-hosted font files from node_modules
npm run build # node build.mjs ŌĆö src/content + src/templates -> dist/ (20 static HTML pages)
npm run serve # node serve.mjs ŌĆö tiny zero-dependency static server at :4550
npm run dev   # build + serve
```

`docs-app/dist` is generated output (gitignored, like the core's `dist/`) ŌĆö never hand-edit files there, edit `docs-app/src/` and rebuild. `docs-app/src/assets/fonts/*.woff2` are also gitignored (regenerated by the `postinstall` script from the `@fontsource-variable/*` packages). `docs-app/**` is excluded from the root `eslint.config.js` for the same reason `examples/**` is.

### CI, release, and governance

- **`.github/workflows/ci.yml`** ŌĆö runs on every push to `main` and every PR: `lint` job (ESLint + Prettier check, single Node version), `test` job (`npm run test:coverage`, matrixed across Node 18/20/22 ŌĆö 18 is the `engines` floor, 22 is current), `build` job (`npm run build`).
- **`.github/workflows/commitlint.yml`** ŌĆö validates PR commit messages against Conventional Commits (`commitlint.config.js` extends `@commitlint/config-conventional`) using `wagoid/commitlint-github-action`. This is CI-only by design ŌĆö there is deliberately no local `husky` git hook, so it never blocks a local commit, only a PR.
- **`.github/workflows/release.yml`** ŌĆö on push to `main`, runs `changesets/action`: if there are pending changesets it opens/updates a "Version Packages" PR (bumps version, updates `CHANGELOG.md`); merging that PR triggers the same workflow to actually `npm publish` (via the root `release` script, which runs `build` then `changeset publish`), with npm provenance (`permissions.id-token: write` + `publishConfig.provenance: true` in `package.json`). **This requires an `NPM_TOKEN` secret configured in the GitHub repo settings** (an npm automation token with publish rights) ŌĆö without it the publish step will fail; this was a deliberate manual step left to the repo owner rather than something set up here.
- **Changesets** (`.changeset/config.json`, `.changeset/*.md` pending-change files) is the single source of truth for versioning and `CHANGELOG.md` ŌĆö see `CONTRIBUTING.md`'s "Changesets (versioning)" section for the contributor-facing flow (`npx changeset`). The repo is single-package (no npm workspaces), so changesets only ever version the root `tinydi` package; `examples/**` and `docs-app/**` are untouched by it.
- **`typedoc.json`** generates a standalone API reference from the same JSDoc already required on every public member (`npm run docs:api` ŌåÆ `api-docs/`, gitignored). It is intentionally _not_ wired into `docs-app`'s own hand-built static-site generator/design system ŌĆö that would mean reconciling two very different HTML outputs (TypeDoc's own theme vs. `docs-app`'s Vite-like dark design) for a phase-4 bonus item; it's a separate, unstyled reference output for now.
- Governance docs (`CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, `CHANGELOG.md`) and GitHub templates (`.github/ISSUE_TEMPLATE/*.yml`, `.github/pull_request_template.md`) follow the same "no real domain/contact yet" placeholder pattern as `docs-app`'s `SITE_URL` (`https://tinydi.dev`) ŌĆö e.g. `CODE_OF_CONDUCT.md`'s enforcement contact uses the same placeholder domain, and `SECURITY.md` routes reports through GitHub's private vulnerability reporting instead of an email address, to avoid inventing one.

## Architecture

Everything lives in `src/`, one concern per file, wired together in `src/index.ts` (the public API surface ŌĆö anything not exported there is internal):

- **`token.ts`** ŌĆö `createToken<T>(description)` returns a `Token<T>` keyed by a `symbol` (identity), not by its `description` string (which is only a debug label ŌĆö two tokens with the same description are distinct registrations). The type `T` is carried via an optional `__type?: T` phantom property that is never actually assigned at runtime; this is what lets `container.resolve(token)` infer the return type with no explicit generic. When touching `Token`, preserve this: don't add a real runtime-assigned field for `T`.
- **`service-lifetime.ts`** ŌĆö `ServiceLifetime` enum, only `Singleton` (default) and `Transient`. **Do not add `Scoped` here** ŌĆö it's an intentional future extension, not in scope for the current version (see `task/01_core_library.md` for the reasoning).
- **`errors.ts`** ŌĆö error hierarchy: abstract `ContainerError` (base, carries the involved `Token`) ŌåÆ `RegistrationError`, `ResolutionError`, `CircularDependencyError`. `CircularDependencyError` renders the full cycle path as `A\n -> B\n -> C\n -> A`; that exact format is asserted in tests, so if you touch `CircularDependencyError.formatPath`, keep the first token unprefixed and every subsequent one prefixed with `' -> '`, including the repeated closing token.
- **`container.ts`** ŌĆö the `Container` class. Key internal invariants:
  - Registering the same token twice (`registerInstance` or `registerFactory`, in any combination) throws `RegistrationError`. This is a deliberate design choice (not obvious from the task spec) made to give `RegistrationError` an actual purpose ŌĆö see `reports/01-core-library.md` for the rationale. To replace a registration, callers must `remove()` first (or `clear()`).
  - Factories have the signature `(container: Container) => T` ŌĆö the container is passed explicitly to the factory (not captured via closure) so dependency wiring stays visible and so the container can track the resolution path.
  - Circular dependency detection works via an internal `resolutionPath: Token<unknown>[]` stack pushed/popped around each factory invocation; when a token already on the stack is requested again, it throws `CircularDependencyError` with the stack plus the repeated token.
  - Singleton factories cache their built instance on the internal registration record (`cachedInstance`/`hasCachedInstance`); Transient factories never cache.

`examples/` holds 7 independent demo packages (`basic`, `service`, `repository-pattern`, `node-backend`, `vue`, `nuxt`, `react`), each with its own `package.json`/`tsconfig.json`/README. In every framework example (`vue`, `nuxt`, `react`) the DI composition root (`di/container.ts` or `services/*.ts` + `plugins/di.ts`) stays framework-agnostic ŌĆö it never imports Vue/Nuxt/React ŌĆö and a thin, framework-idiomatic adapter (`di/plugin.ts` for Vue, `plugins/di.ts` + `composables/useService.ts` for Nuxt, `di/DIProvider.tsx` for React) is the only place that bridges the container into that framework's own DI-like mechanism (`provide`/`inject`, Nuxt plugin `provide`, React Context). Keep that separation when touching these examples.

`docs-app/` is a hand-written static-site generator (no 11ty/Astro/etc.) that turns structured content into the plain HTML/CSS/JS site actually shipped ŌĆö see `docs-app/README.md` for the full file-by-file breakdown. The parts worth knowing before touching it:

- **Content is data, not markup.** Each doc page is `docs-app/src/content/{en,it}/docs/<id>.mjs`, exporting `meta` (title/description) and `blocks` ŌĆö a small DSL (`heading`/`p`/`list`/`code`/`callout`/`api-table`/`compare-table`/`example-grid`/`faq-list`) rendered by `src/lib/blocks.mjs`. That same renderer extracts the search-index chunks (one per h2/h3/FAQ item) from the identical source, so content and search index can never drift apart. `docs-app/src/nav.mjs` is the one place that lists the 9 doc pages and every UI string per language ŌĆö add a new doc page there first.
- **No shipped runtime dependency does syntax highlighting or search.** `src/lib/highlight.mjs` is a hand-rolled regex tokenizer (ts/bash/json) run at _build_ time, emitting already-colored `<span>`s baked into the static HTML (no client-side highlighting JS, no FOUC). `src/scripts/search.js` fetches the prebuilt `search-index.{en,it}.json` and scores matches with a small hand-written substring scorer ŌĆö no Fuse.js or similar.
- **The scroll-spy ("On this page") is a plain scroll listener, not `IntersectionObserver`.** `src/scripts/nav.js` finds the last heading scrolled past (rAF-throttled), so a TOC entry stays active for its whole section rather than only while its heading is on screen. Also: `IntersectionObserver`'s `rootMargin` only accepts `px`/`%`, never `rem` ŌĆö it throws a silent-in-the-UI `SyntaxError` if you get that wrong (this shipped broken once; only caught by checking the browser console, not by reading the code).
- **Language switching never auto-redirects.** The chosen language persists in `localStorage`, but a mismatched page only shows a dismissible banner ("Continue in English"/"Continua in italiano") ŌĆö never a forced redirect (bad for deep links, back-navigation, and `hreflang`-evaluating crawlers). See `reports/03-docs-website.md` for the reasoning.
- **Any visual state that depends on `localStorage`/`sessionStorage`/`navigator.*` must be decided by an inline, synchronous script in `<head>`, before first paint** ŌĆö never by a `defer`red script. `templates/layout.mjs` has three of these (`themeInitScript`, `platformInitScript`, `langBannerInitScript`), each setting a `data-*` attribute on `<html>` that CSS then reads (e.g. `:root[data-lang-banner-show] [data-lang-banner] { display: block; }`). This shipped broken once for the language banner: it used to be decided in the deferred `scripts/lang.js`, which only runs after the page has already rendered, so readers with a stored language preference saw the banner pop in and shove the whole page down a beat after every load/navigation (only caught by reproducing it in the browser with a stored preference set, not by reading the code). `scripts/lang.js` now only handles persisting the choice and the dismiss button.
- **The one piece of bespoke imagery** is `src/lib/graph-diagram.mjs`, a hand-built SVG of the actual resolve flow (Token ŌåÆ Container ŌåÆ Factory ŌåÆ Instance), reused unlabeled in the hero and labeled in the homepage Architecture section ŌĆö deliberately not a stock photo.
- Every icon SVG needs an explicit size somewhere in `src/styles/components.css` (a class targeting it or its parent) ŌĆö `base.css` has a defensive fallback (`svg { width: 1em; height: 1em; }`) precisely because a missing size rule once rendered a raw icon at its ~300├Ś150 intrinsic size instead of failing loudly.

## Design decisions to preserve

- No dependency arrays in `package.json` ŌĆö only `devDependencies` (tooling) and an optional `peerDependencies.typescript: ">=6.0.0"`. Don't introduce a runtime dependency without discussing it first; "zero external dependencies in the core" is a hard constraint from the project spec.
- Minimum supported TypeScript is **6.0** (`devDependencies.typescript` is pinned to `^6.0.3`, capped below the range `typescript-eslint@8.65.0` supports, which is `>=4.8.4 <6.1.0`; `typedoc@0.28.x`'s peer range, `5.0.x`ŌĆō`6.0.x`, is the other current ceiling). If bumping TypeScript past 6.1, check both `typescript-eslint` and `typedoc` compatibility first.
- The published package is a genuine dual ESM+CJS build via `tsup` (not a bundler-free `tsc` emit) ŌĆö `package.json`'s `exports` map has separate `import`/`require` conditions, each with their own `types` entry (`dist/index.d.ts` for ESM, `dist/index.d.cts` for CJS), matching what `tsup` actually emits. Keep both conditions in sync if the entry point ever changes.
- Every public API member needs full JSDoc (signature, params, return, a usage `@example`, relevant edge cases) ŌĆö this is enforced by convention, not by a lint rule, so check for it in review.
- `eslint.config.js` and `vitest.config.ts` are deliberately excluded from typed linting (`*.config.js`/`*.config.ts` override with `disableTypeChecked`) because they aren't covered by `tsconfig.json`'s `include`. `tsconfig.json` (base, no `rootDir`) covers `src` + `tests` for editor/lint/type-check; `tsconfig.build.json` adds `rootDir: "src"` and excludes tests, and is what `npm run build` actually uses to emit `dist/`.
- `examples/**` is in the root `eslint.config.js` `ignores` list. Without it, `npm run lint`'s `projectService: true` tries to type-check every example's own `dist`/`.output`/`.nuxt` build artifacts too (they aren't covered by any `tsconfig.json`), producing hundreds of parse errors unrelated to the core.
- Coverage target is >95%; the current suite is at 100% on all `src/*.ts` (excluding `index.ts`, which is pure re-exports).
- The root `package-lock.json` is committed (as of Phase 4) ŌĆö it used to be gitignored, but `.github/workflows/*.yml`'s `actions/setup-node@v4` (`cache: npm`) and every `npm ci` step need a real, checked-out lockfile to work at all; without one, every CI/release job fails immediately at the `setup-node` step. Same reproducibility rationale as `examples/*`'s and `docs-app`'s already-committed lockfiles.

## Repository layout notes

- `task/` and `reports/` exist locally but are **gitignored** (`/task`, `/reports` in `.gitignore`) ŌĆö they hold the phased project spec and the end-of-phase reports and are intentionally not pushed to the remote. They still exist on disk in this working directory and are worth reading for context (e.g. `task/01_core_library.md`/`task/02_examples_and_readme.md`/`task/03_docs_website.md` for the original spec and constraints, `reports/01-core-library.md`/`reports/02-examples-and-readme.md`/`reports/03-docs-website.md` for what was actually built, problems hit, and scope trade-offs), but don't assume a fresh clone of the repo will have them.
- The task spec describes one further phase not yet implemented: an internal blog + closing write-up (`task/05_...`). The package has never been published to npm yet ŌĆö `README.md`/`README.it.md`/`docs-app` all document `npm install tinydi` as the intended future install path; until the first real `npm publish` happens (see "CI, release, and governance" above ŌĆö it needs an `NPM_TOKEN` secret the repo owner adds manually), consume the local build via `file:` (as `examples/` does) or `npm link`. `PRODUCT.md` and `DESIGN.md` at the repo root are the Impeccable-skill strategic/visual-design source of truth behind `docs-app/` (register: brand; Vite-like dark-first direction, one accent, self-hosted fonts) ŌĆö read them before changing the site's visual language. `docs-app/src/nav.mjs`'s `SITE_URL` is a placeholder domain (`https://tinydi.dev`) used for `hreflang`/canonical/sitemap URLs until a real domain exists.
